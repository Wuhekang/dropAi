# 中文学术轻量降 AI Skill

## 目标

将中文学术正文改写得更接近真实作者的自然修改痕迹：保留论文语气，减少模板化、机械化和过度工整表达。

本 Skill 面向学术写作质量优化，不承诺规避任何检测。优先保证原意稳定、文本自然、不过度扩写。

## 适用范围

只处理连续正文段落。遇到以下内容时，应尽量原样保留，不要改成解释性说明文：

- 代码、注解、接口路径、配置项、SQL、表字段、变量名、类名、方法名。
- 数据库表结构、字段名称、字段类型、默认值、主键、外键。
- 公式、枚举值、文件路径、URL、命令、JSON、XML。
- 已经足够自然、没有明显模板化痕迹的短段落。

## 硬约束

1. 必须保留原意、论点、事实和逻辑关系。
2. 不得新增原文没有的数据、案例、引用、作者、年份、系统功能或结论。
3. 不得输出标题、列表、多个版本、解释过程、说明文字或“以下是”。
4. 不得把代码、字段、接口、SQL 或变量解释成完整说明文。
5. 不得使用“首先、其次、最后、综上所述、值得注意的是、由此可见、随着……的发展、具有重要意义”等模板化表达。
6. 不得连续三句使用相同结构。
7. 不得把文本改得更正式、更空泛或更像政策报告。
8. 不得堆叠“内在机制、实践启示、有效路径、现实意义、系统性分析、优化效果、用户体验提升”等泛化套话。
9. 不得只做同义词替换；但也不要为了降重而大幅扩写。
10. 改写后长度通常控制在原文的 85% 到 115% 之间。原文为短句、代码说明或功能点时，尽量不超过原文长度。

## 改写策略

- 采用“轻量改写”：优先调整语序、删减模板词、改变局部句式，而不是整段重写。
- 保留原文中的专业名词、技术名词、模块名称、字段名称和方法名称。
- 减少显性连接词。能靠语义自然承接时，不添加“因此、此外、同时、然而、从而、进而”。
- 保留长短句差异，不要把所有句子改成相近长度。
- 对功能描述类段落，保持简洁，不把每个功能展开成“全流程、全过程、多维度”等泛化表达。
- 对系统设计类论文，允许保留朴素、直接的工程表达；不要把技术描述改成泛泛的管理学表达。
- 如果原文已经自然，只做少量顺序调整或删减，不强行改写。

## 双降模式

当优化类型为“双降”时，不要分成“先大幅降重、再降 AI”的两种写法。必须在同一版输出里同时满足：

- AI 风险优先：宁可少降一点重复，也不要把文本改得更工整、更完整、更像机器生成。
- 降重方式以局部处理为主：调整语序、替换少量短语、拆合个别句子、删除冗余限定词。
- 禁止为了降重扩写功能点，尤其不要把“员工管理、绩效管理、考勤管理”等枚举展开成大段解释。
- 禁止新增“全流程、全周期、多维度、一体化、系统性、有效提升、优化效果”等泛化词。
- 改写后通常不超过原文 110%。原文本身较短时，只做轻改。

## 质量自检

输出前自检：

- 是否比原文更长、更完整、更工整？如果是，压缩。
- 是否新增了“全流程、多维度、系统性、有效提升、优化效果”等泛化词？如果是，删除。
- 是否把代码、字段或接口说明改成解释性长句？如果是，恢复为接近原文的技术表达。
- 是否仍含模板连接词？如果是，删除或改成自然承接。
- 是否改变核心含义？如果是，恢复原意。

## 输出

只输出最终改写后的正文段落，不输出任何解释。
